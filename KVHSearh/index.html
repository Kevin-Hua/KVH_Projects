01 / overview

專案簡介Project Overview

KVHSearch 是一套為大型韌體／BIOS Git 倉庫量身打造的離線全文索引搜尋系統。 它以 SQLite 的 FTS5 虛擬表（Trigram tokenizer）作為倒排索引後端， 並在 Python 層提供 CLI 介面與 GUI 介面，讓工程師能在毫秒級完成跨越萬支檔案的關鍵字搜尋， 而不需要依賴網路、伺服器或 Elasticsearch 等重量級工具。

KVHSearch is an offline full-text trigram search system purpose-built for large UEFI/BIOS Git repositories. It uses SQLite FTS5 (Trigram tokenizer) as the inverted-index backend and exposes both a CLI and a GUI, enabling engineers to search across tens of thousands of files in milliseconds—no network, no server, no Elasticsearch required.

04 / algorithm

索引建立演算法Index Build Algorithm

Trigram 倒排索引Trigram Inverted Index

SQLite FTS5 的 tokenize = 'trigram' 會將每個字串切成連續的 3 字元滑動視窗 token， 並為每個 token 建立倒排清單（指向包含該 token 的 row）。

SQLite FTS5's tokenize = 'trigram' slices every string into consecutive 3-character sliding-window tokens and builds an inverted list for each token (pointing to the rows that contain it).

06 / tiered index design

分層索引設計Tiered Index Design

分層索引是 KVHSearch 最核心的設計，靈感來自 LSM-tree 和傳統備份的「完整備份 + 增量備份」概念。

Tiered indexing is KVHSearch's most central design, inspired by LSM-trees and the classic backup concept of "full backup + incremental backup".

── 版本化分層結構示意（帶 tombstone）

Tombstone（刪除標記）機制Tombstone (Deletion Marker) Mechanism

當 git diff 顯示某檔案被刪除（status = D）或重新命名（status = R 的舊路徑）， 該路徑會被寫入 minor DB 的 tombstones 表。 搜尋時，_run_query() 會用 tombstone set 過濾掉 main 查詢結果中的對應記錄。

When git diff shows a file deleted (status D) or renamed (the old path of status R), that path is written to the minor DB's tombstones table. At search time, _run_query() uses the tombstone set to suppress matching records from the main results.

⚠️
重要：Minor 是以 Main 為基準的完整差量 Minor_v07_05 包含的是「從 V07 base 到 minor_05 HEAD 的所有差異」， 不是「minor_04 到 minor_05」的一步差異。 這意味著每次只需合併 Main + 當前 Minor，不需要疊加多個 minor。 Important: Each Minor is a full delta relative to Main Minor_v07_05 contains "all differences from the V07 base to minor_05 HEAD", not just the one-step delta from minor_04 to minor_05. This means only Main + current Minor need to be merged—no stacking of multiple minors.
X / change detection

變更偵測策略Change Detection Strategy

在索引建立過程中，_process_file() 並非對所有檔案一律重新讀取與重建索引， 而是採用 由快到慢的三層變更偵測策略， 盡可能在最早階段判定「檔案未變更」並直接跳過， 以大幅降低磁碟 IO 與 CPU 計算成本。

During index building, _process_file() does not unconditionally re-read and re-index every file. Instead it applies a three-tier change-detection strategy (fast→slow), determining "file unchanged" as early as possible and skipping immediately to minimise disk I/O and CPU cost.

三層偵測順序（由快至慢）Three-tier Detection Order (Fast to Slow)

    ① Git Blob SHA 比對（零讀檔） 對於 git tracked 的檔案，優先透過 git ls-files -s 取得的 blob SHA1 與 DB 中儲存的 blob_sha 進行比對。 若 SHA 相同，代表檔案內容完全一致， 可直接判定為「未變更」，不需讀取檔案本體。 ① Git Blob SHA Compare (zero file reads) For git-tracked files, the blob SHA1 from git ls-files -s is compared against the stored blob_sha in the DB. An identical SHA means the file content is unchanged—skip immediately, no file body read required.
    ② mtime 修改時間比對（metadata only） 若檔案未被 git 追蹤，則使用檔案系統的 mtime 與前次索引紀錄比較。 當時間差小於安全閾值（例如 < 0.01 秒）， 視為未變更，避免不必要的內容讀取。 ② mtime Modification-time Compare (metadata only) For untracked files, the filesystem mtime is compared against the previously recorded value. If the difference is below the safety threshold (e.g. < 0.01 s), the file is considered unchanged—no content read needed.
    ③ BLAKE2b 雜湊比對（完整讀檔） 僅在前兩層無法確認時， 才讀取完整檔案並計算 BLAKE2b(digest_size=20) 雜湊。 若雜湊一致，代表內容實際未改變； 否則視為修改或新增，進行索引更新。 ③ BLAKE2b Hash Compare (full file read) Only when the first two tiers are inconclusive is the full file read and BLAKE2b(digest_size=20) computed. An identical digest means content is truly unchanged; otherwise the file is treated as modified or new and the index is updated.
⚡
效能效果 在實際大型 Git 倉庫中， 多數日常 build 僅有少量檔案變更， 三層策略通常可跳過 95% 以上的檔案， 使增量索引時間從分鐘級縮短至數秒。 Performance Impact In real large Git repositories, most daily builds change only a handful of files. The three-tier strategy typically skips 95%+ of files, reducing incremental index time from minutes to seconds.

Fuzzy 演算法詳解（刪除 + 替換展開）Fuzzy Algorithm (Deletion + Substitution)

_fuzzy_variants_dist1() 在 dist=1 時同時展開刪除（Deletion）與替換（Substitution）兩種鄰域； dist≥2 時 _pigeonhole_parts() 以鴿巢原理分段，再透過 _bk_tree_fuzzy() 以 FTS5 OR MATCH 批次查詢；最後 Phase 2 由 _scan_lines_fuzzy() 以 _levenshtein() 逐行精確驗證。

At dist=1, _fuzzy_variants_dist1() expands both Deletion and Substitution neighbourhoods. At dist≥2, _pigeonhole_parts() segments the keyword via the Pigeonhole principle; _bk_tree_fuzzy() then batch-queries with FTS5 OR MATCH; Phase 2 uses _scan_lines_fuzzy() + _levenshtein() for exact per-line verification. 

# 範例：keyword="OemDxe", max_dist=1
# 刪除集合（6 個）：
# "emDxe", "OmDxe", "OeDxe", "Oemxe", "OemDe", "OemDx"
# 替換集合（每位置 × _FUZZY_ALPHABET 36 字元）：
# "AemDxe", "BemDxe", ..., "OemDxa", "OemDxb", ...
# → 合入 FTS5 MATCH 的 OR 查詢，只保留長度 ≥ 3 的候選詞
# Phase 2：_scan_lines_fuzzy() 以 _levenshtein() 驗證每行各 token

Pigeonhole 分段策略（dist ≥ 2）Pigeonhole Segmentation (dist ≥ 2)

_pigeonhole_parts() 將關鍵字切成 d+1 個等段。 根據鴿巢原理，若 edit_distance(query, target) ≤ d，則至少有一段原封不動出現在 target， 可直接作為 FTS5 子字串過濾，大幅縮小候選集後再交 Levenshtein 精確驗證。

_pigeonhole_parts() splits the keyword into d+1 equal segments. By the Pigeonhole principle, if edit_distance(query, target) ≤ d, at least one segment must appear verbatim in the target—that segment is used as a FTS5 substring filter to shrink the candidate set before Levenshtein exact verification.

複合 C 運算式解析Compound C Expression Parsing

search_fuzzy() 支援直接輸入 C 複合運算式作為關鍵字。 _parse_structure() 解析 ->、.、( 分隔符， 拆出識別符 parts 與結構 constraints；Phase 2 的 _scan_lines_fuzzy() 同時驗證 Levenshtein 距離與結構順序（_constraint_to_regex()）。

search_fuzzy() accepts C compound expressions directly as the keyword. _parse_structure() splits on ->, ., ( separators to extract identifier parts and structural constraints; Phase 2's _scan_lines_fuzzy() verifies both Levenshtein distance and structural order (_constraint_to_regex()). 

# 複合運算式範例
python Search_cli.py fuzzy "gProtocol->IsSupported" 1
# → parts: ['gProtocol', 'IsSupported']
# → constraint: ('gProtocol', '->', 'IsSupported')
# 只輸出同時 fuzzy 命中兩個 token 且 '->' 結構正確的行

python Search_cli.py fuzzy "PcdGet32(PcdDebugLevel)" 1
# → parts: ['PcdGet32', 'PcdDebugLevel']
# → constraint: ('PcdGet32', '(', 'PcdDebugLevel')
💡
Fuzzy 目前的限制 支援刪除與替換（dist=1 展開）；不支援插入或轉置。 dist=2 以上藉 Pigeonhole 加速候選集縮減，Phase 2 用標準 Levenshtein 驗證， 所有 ≤ max_dist 的編輯距離皆能正確覆蓋。 Current Fuzzy Limitations Supports deletion and substitution (dist=1 expansion); insertion and transposition are not supported. For dist≥2, Pigeonhole accelerates candidate reduction; Phase 2 uses standard Levenshtein—all edit distances ≤ max_dist are correctly covered.

10 / merge logic

Tiered 合併邏輯Tiered Merge Logic

_run_query() 實作 Minor 與 Main 結果的合併，規則如下：

_run_query() implements the merge of Minor and Main results according to the following rules: 

def _run_query(primary, secondary, tombstones, sql, params):
    rows = primary.execute(sql, params).fetchall()   # minor 結果
    if secondary:
        seen = {r[0] for r in rows}             # minor 中已有的 path
        for r in secondary.execute(sql, params).fetchall():
            if r[0] not in tombstones and r[0] not in seen:
                rows.append(r)                       # 補充 main 的獨有結果
    return rows

# 在 kvhsearch_core.py 的最上層修改閾值
AUTO_BUILD_THRESHOLD: int = 20   # 0 = 停用自動建立

超過閾值或自動建立失敗時，系統退回 Main-only 模式並顯示黃色警告。 此設計是為了避免在 commits 很多時，自動建立花太多時間拖慢搜尋回應。

When the threshold is exceeded or auto-build fails, the system falls back to Main-only mode and shows a yellow warning. This is intentional—preventing auto-build from delaying search response when there are many commits ahead.

15 / limitations

限制Limitations

  • Fuzzy 不支援插入與轉置Fuzzy: No Insertion or Transposition：search_fuzzy() 支援刪除與替換（dist=1 展開），但不支援插入（"OemDxe" → "OemDxes"）或轉置（"OemDxe" → "OmeDxe"）；dist≥2 時轉置錯誤通常仍可被覆蓋。search_fuzzy() supports deletion and substitution but not insertion ("OemDxe"→"OemDxes") or transposition ("OemDxe"→"OmeDxe"). Transpositions at dist≥2 are usually still covered.
  • Minor 不是真正的增量鏈Minor Not a True Incremental Chain：每個 Minor 都是對 Main 的完整差量，不支援 Minor→Minor 的鏈式增量。換言之，Minor_v07_09 不能基於 Minor_v07_08 建立，必須重新 diff Main。Each Minor is a full diff against Main—no Minor→Minor chaining. Minor_v07_09 cannot be based on Minor_v07_08; must re-diff Main.
  • HEX 索引會大幅增加 DB 大小HEX Indexing Inflates DB Size：對大型二進位倉庫全面啟用 --hex 會讓 DB 膨脹數倍，需謹慎評估。Fully enabling --hex on large binary repos can inflate the DB severalfold.
  • 不支援正規表示式（Regex）搜尋No Regex Search：FTS5 Trigram 模式不提供 regex 查詢，只能透過 --grep 做 post-filtering。FTS5 Trigram does not support regex queries. Use --grep for post-filtering instead.
  • 不支援多行 grepNo Multi-line Grep：--grep 篩選是逐行比對，無法匹配跨行的模式（如函式定義跨多行）。--grep filters line-by-line; cannot match patterns spanning multiple lines (e.g. multi-line function definitions).
  • SQLite 並發寫入受限SQLite Concurrent Write Limit：WAL 模式雖允許多讀一寫，但同時只能有一個 build 程序寫入同一個 DB，不支援分散式建立。WAL mode allows multiple readers but only one build process can write to the same DB. Distributed builds are not supported.
  • 需要 git 才能使用分層功能Git Required for Tiered Features：build_minor、build_chain 等功能完全依賴 git CLI，非 git 倉庫只能使用 build 單一索引模式。build_minor, build_chain etc. depend entirely on git CLI. Non-git repos can only use the single build mode.
  • 64 MB 檔案大小上限64 MB File Size Limit：預設跳過超過 64 MB 的檔案（可在 JSON 設定 max_file_mb 調整），超大型二進位檔無法索引。Files over 64 MB are skipped by default (configurable via max_file_mb in JSON). Very large binaries cannot be indexed.
  • 自動建立 minor 有延遲Auto-build Minor Has Latency：第一次搜尋時觸發自動 build_minor，會有數秒到數十秒的延遲，後續搜尋才恢復毫秒級。First search on a new branch triggers auto build_minor, causing a delay of seconds to tens of seconds. Subsequent searches return to millisecond speed.
  • 索引不含行號資訊Index Does Not Store Line Numbers：FTS5 content 欄位只存完整文字，行號是搜尋時即時從原始檔讀取的，若原始檔被移動或刪除，行號顯示會失效。FTS5 content column stores full text only. Line numbers are read from the source file at query time; if the file is moved or deleted, line numbers will be invalid.
  • 不支援符號語意搜尋No Semantic Symbol Search：無法理解「這個函式呼叫了哪些函式」或「這個型別的所有用法」，只能做字串層面的搜尋。Cannot understand "what functions does this call" or "all usages of this type"—string-level search only.
X / advanced risks

進階限制與風險評估Advanced Risks & Limitations

  • 索引 DB 的生命週期管理需要人為介入： 當啟用 build_history、長期保留多個 main 版本， 或對二進位檔大量啟用 --hex 時， SQLite DB 會持續累積，磁碟空間與管理成本上升。 目前系統提供 cleanup、build_promote 等工具， 但尚未內建「保留策略」（例如僅保留最近 N 個 main 或以容量為門檻的自動清理）。
  • search_fuzzy 的結果一致性依賴使用者理解其模型： Fuzzy 搜尋支援刪除與替換兩種展開，對「少打或打錯一個字」均有效， 但不支援插入或轉置；dist≥2 進一步透過 Pigeonhole 加速，Phase 2 以 Levenshtein 精確驗證。 複合運算式（A->B）亦能驗證結構順序，但仍建議僅將 fuzzy 作為「輔助縮小範圍」的工具，而非唯一搜尋手段。
  • 第一次自動 build_minor 可能造成搜尋延遲體感： 在 auto-build 啟用的情況下， 使用者於新分支首次搜尋時，系統可能觸發 build_minor()， 導致搜尋回應從毫秒級延遲至數秒甚至數十秒。 此行為屬刻意設計的效能取捨， 但在 GUI 使用情境中可能需要額外提示以避免誤解為「程式卡住」。
  • GUI 層為 Thin Wrapper，功能與 CLI 完全對齊： 目前 GUI 著重於搜尋體驗，並未實作進階索引管理（如 build_chain、cleanup、promote）。 在多人共用或非工程背景使用者場景中， 仍需搭配 CLI 或預先建立索引， GUI 並非完整管理介面，而是核心引擎的輕量視窗入口。
  • FTS5 Trigram 對極短關鍵字仍需退回線性掃描： 長度小於 3 的子字串搜尋會退回 LIKE， 在查詢頻繁且範圍廣的情況下，可能造成效能下降。 這是 trigram tokenizer 的先天限制， 而非 KVHSearch 的實作問題。
  • 索引系統為「內容導向」，非「語意導向」： KVHSearch 僅對檔案內容與路徑建立倒排索引， 不理解語言結構、函式邊界、型別關係或呼叫圖。 對需要語意分析的需求（例如「誰呼叫了這個 API」）， 仍需搭配 ctags、cscope 或 LSP 類工具使用。

Tombstones). They are merged at query time, reducing rebuild time from minutes to seconds in fast-switching branch environments.
三層跳過策略最大化增量效能 優先用 git blob SHA（零讀檔）→ mtime（metadata only）→ BLAKE2b（完整讀取）， 在日常重建中可跳過 95%+ 的未修改檔案。 Three-tier Skip Strategy Maximises Incremental Performance git blob SHA (zero reads) → mtime (metadata only) → BLAKE2b (full read). In daily rebuilds, 95%+ of unchanged files are skipped.
